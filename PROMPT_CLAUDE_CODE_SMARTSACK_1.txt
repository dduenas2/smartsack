## CONTEXTO DEL PROYECTO

SmartSack es una plataforma web inteligente para gestión de producción en plantas de fabricación de sacos de papel. Es mi trabajo de grado en Ingeniería de Software (Politécnico Grancolombiano).

## LOS 3 PILARES

1. **Digital Twin** de la línea de producción:
   - Vista operario: estado de SU máquina, orden en proceso, avance %, cola de órdenes, botones de registro rápido (paradas, cambios de formato, incidencias)
   - Vista supervisor: mapa completo de planta con TODAS las máquinas codificadas por color según estado, actualización en tiempo real vía WebSockets

2. **Motor de ML predictivo**:
   - Predicción de retrasos en órdenes de producción
   - Algoritmos: Random Forest y XGBoost
   - Dashboard de KPIs: OEE (Disponibilidad × Rendimiento × Calidad), tasa de cumplimiento, tendencias
   - Alertas proactivas cuando una orden tiene alta probabilidad de retrasarse

3. **Chatbot IA conversacional**:
   - API de Claude (Anthropic) orquestada con LangChain
   - Function calling para traducir preguntas en lenguaje natural a consultas SQL sobre PostgreSQL
   - Ejemplos: "¿Cuántos sacos de cemento se produjeron ayer en turno 2?", "¿Qué órdenes están retrasadas hoy?"

## STACK TECNOLÓGICO (NO CAMBIAR)

**Frontend:** React 18 + JavaScript (NO TypeScript) + Vite + Recharts + Tailwind CSS
**Backend:** Python 3.11+ + FastAPI + SQLAlchemy + Pydantic + PyJWT + WebSockets (Starlette)
**Base de datos:** PostgreSQL 16 + Redis (caché de estado de máquinas)
**ETL:** Pandas (CSV del ERP → PostgreSQL)
**ML:** Scikit-learn + XGBoost + Matplotlib + Seaborn + Jupyter Notebook + Joblib
**IA Generativa:** API de Claude (Anthropic) + LangChain + function calling
**Infraestructura:** Docker + Docker Compose + Nginx (reverse proxy) + Git + GitHub + Pytest

## ARQUITECTURA DE DATOS

Doble fuente:
1. **ETL por lotes**: CSVs exportados del ERP (SAP) → Pandas → PostgreSQL
   - Transacciones: órdenes de producción, confirmaciones, movimientos de material, despachos
   - Validación automática, detección de duplicados, log de errores
2. **Captura en tiempo real**: Operarios registran eventos desde el PC de su máquina
   - Paradas, cambios de formato, incidencias, fin de órdenes
   - Se envía al backend vía API REST y se propaga a supervisores vía WebSockets

## MODELO DE DATOS (diseña estas tablas mínimo)

- **users**: id, username, password_hash, role (operario/supervisor/admin), machine_id (nullable), created_at
- **machines**: id, name, code, type, location, status (running/stopped/maintenance/idle), current_order_id
- **production_orders**: id, order_number, product_type, product_description, quantity_ordered, quantity_produced, machine_id, status (pending/in_progress/completed/delayed), priority, planned_start, planned_end, actual_start, actual_end, created_at
- **production_events**: id, machine_id, order_id, user_id, event_type (start/stop/pause/format_change/incident/end), description, timestamp
- **shifts**: id, name (turno_1/turno_2/turno_3), start_time, end_time
- **materials**: id, order_id, material_type, quantity_planned, quantity_used
- **quality_records**: id, order_id, total_produced, good_units, defective_units, timestamp
- **oee_records**: id, machine_id, shift_id, date, availability, performance, quality, oee_value
- **ml_predictions**: id, order_id, delay_probability, predicted_delay_hours, features_json, model_version, created_at

## REGLAS DE DESARROLLO

1. **Comentarios**: Cada archivo debe tener un docstring al inicio explicando su propósito. Funciones complejas deben tener docstrings.
2. **Código limpio**: Nombres descriptivos en inglés para código, pero la interfaz de usuario en español.
3. **Git**: Commits descriptivos en español. Un commit por cada funcionalidad completada.
4. **Variables de entorno**: Toda configuración sensible en .env (nunca hardcodeada).
5. **Error handling**: Try/except con logging apropiado. Nunca fallar silenciosamente.
6. **Validación**: Pydantic para validar toda entrada de datos en la API.
7. **Documentación Swagger**: FastAPI la genera automáticamente, pero agrega descripciones a cada endpoint.
8. **Datos de ejemplo**: Genera datos de prueba realistas para desarrollo (seed data).

## DATOS DE EJEMPLO REALISTAS

Para desarrollo, genera datos que simulen una planta de sacos de papel real:
- 8 máquinas: Tubuladora 1, Tubuladora 2, Impresora 1, Impresora 2, Fondadora 1, Fondadora 2, Empacadora 1, Empacadora 2
- 3 turnos: Turno 1 (06:00-14:00), Turno 2 (14:00-22:00), Turno 3 (22:00-06:00)
- Tipos de producto: Saco cemento 50kg, Saco cemento 25kg, Saco harina 50kg, Saco fertilizante 25kg, Saco cal 25kg
- Órdenes con cantidades entre 5,000 y 50,000 unidades
- Genera al menos 6 meses de datos históricos para entrenar el modelo de ML
